# Tiny Owl 检测原理白皮书

> 🦉 **挑选靠谱的中转站，从看穿它开始。**
>
> 本文系统介绍 Tiny Owl 猫头鹰评测的技术检测体系：我们如何在**只有一个 API 地址和一把 Key** 的黑盒条件下，识破 API 中转站的「模型身份替换」「降智」「协议造假」等套路。

---

## 引言：中转站的「障眼法」时代

API 中转站（model-relay）以更低价格、更便捷的支付方式，转售 OpenAI、Anthropic、Google 等厂商的大模型能力。便利的背后，是一个用户几乎无法验证的黑箱：

- **模型身份替换**——声称卖你 Opus 4.8，实际后台接的是廉价小模型；
- **降智**——用量化版、蒸馏版冒充满血版，能力悄悄缩水；
- **协议造假 / 高配低卖**——响应字段残缺、缓存复用、掺水限速。

最棘手的是：**这些手段大多发生在「行为层」之下，肉眼根本看不出来。** 你问它「你是谁」，它照样答「我是 Claude」；你让它答题，它也能答对。于是绝大多数检测工具被轻松糊弄。

Tiny Owl 的答案是：**不止听它说什么，更要看它的响应「长什么样」。** 我们用一套七维交叉验证体系，把鉴别的战场从「可以表演的行为层」推进到「难以伪造的协议元数据层」。

## 设计哲学：三层信号，层层加码

学术界对黑盒 LLM API 审计的研究已经给出明确结论：**仅靠文本输出的行为层检测，无法对抗有意作恶的提供方** <sup>[1]</sup>。一个动机充分的中转站，可以在 system prompt 里塞「你是 Claude」，可以让小模型去背标准答案——行为层信号几乎可以任意伪造。

因此 Tiny Owl 把检测信号分为三层，越往下越难伪造：

| 层级 | 信号 | 能否被伪造 | Tiny Owl 策略 |
| --- | --- | --- | --- |
| **行为层** | 自述身份、知识问答、思维链 | 容易（system prompt 即可表演） | 作为基础筛查，但**不单独采信** |
| **统计层** | 输出分布、内容多样性、缓存特征 | 中等（需控制采样行为） | 多轮采样 + 分布检验 |
| **协议层** | 响应 id 前缀、usage 厂商字段、结束原因词表 | 困难（需逐字段重写整套协议） | **最高权重，一票否决** |

> **核心洞察**：行为可以演，但底层响应协议的「厂商签名」极难全部伪造。当一个中转站声称卖 Claude，它的响应却带着 OpenAI 的协议指纹时，再完美的行为表演也会当场穿帮。

下面逐一拆解七个检测维度。

## 七维检测体系

Tiny Owl 在一次检测任务中，对中转站接口发起**多轮、多角度探测**，并行评估七个维度。每个维度独立给出 0–100 子分与「通过 / 可疑 / 未通过 / 无法判定」结论，最终交叉聚合为综合可信度评分。

工程上，我们坚持「**采样（网络 I/O）与判定（纯函数）严格分离**」：所有维度的判定逻辑都是无副作用的纯函数，可被属性测试（property-based testing）穷举验证，杜绝「换个输入就误判」的脆弱性。

### ① 返回协议一致性

校验响应是否真正符合 OpenAI 兼容协议：

- **非流式**：顶层 `id`、`object`、`choices`、`model` 字段是否完备；
- **流式（SSE）**：分块是否为合法的 `data: {...}` 结构、是否有 `[DONE]` 终止符、增量 `delta` 是否规范。

协议残缺往往意味着中转站做了非标准的请求改写或代理拼接，是最基础的「做工」体检。

### ② 响应结构合法性

深入校验响应体的结构字段：

- `usage` 中 `prompt_tokens` / `completion_tokens` / `total_tokens` 是否为非负整数，且 **total = prompt + completion**；
- `finish_reason` 是否落在合法集合；
- `choices[].message.role` 是否为 `assistant`。

`usage` 字段造假（如 total 对不上）是廉价中转的常见破绽。

### ③ 知识问答结果

向接口发送一组**已知确定答案**的探测题，比对返回内容的命中率。命中率过低，说明实际模型能力低于所声称水平——这是判定降智的输入信号之一。

> 我们清楚单靠基准答题无法区分同档模型 <sup>[1]</sup>，因此知识问答在 Tiny Owl 中只作为**降智粗筛**，不承担身份鉴别职责。

### ④ 身份一致性

通过身份探测题解析模型自述身份，与目标模型比对。这里藏着一个**很多检测工具会踩的坑**：

> 正品 Claude 出于安全策略，对「你是哪个模型」这类问题常常**拒答**（例如回复 "I can't discuss that."），而不会主动声称「我是 Claude」。

如果简单地「没说自己是 Claude 就判替换」，**正品反而会被误杀**。Tiny Owl 经真品中转实测校准后采用更严谨的逻辑：

- 自称**竞品厂商**（如目标是 Claude 却自称 GPT）→ 判定失败，触发身份替换警示；
- 自称本厂商，或合规拒答 / 不明示且无竞品信号 → 判定通过。

即：**只有「冒认他人」才算替换，「拒绝表态」不算。**

### ⑤ 思维链痕迹

检测响应中是否存在与目标模型推理特征相匹配的思维链 / 推理痕迹（如显式推理字段、分步推导结构）。对于本身具备强推理能力的目标模型，缺失推理痕迹是能力不符的信号。

### ⑥ 签名指纹（统计层）

对**同一个固定 prompt 重复多次采样**，从输出的统计分布中提取「指纹」。这一思路受黑盒 API 审计研究中**基于秩的均匀性检验（Rank-Based Uniformity Test）**的启发 <sup>[2]</sup>：真实模型在固定输入下的输出，其统计形态具有可区分的特征。

Tiny Owl 在纯文本（无 logprobs）条件下采用稳健的相对统计量：

- **内容去重率**：同一 prompt 多次返回应当内容各异。若多次返回**完全相同**，疑似固定模板或缓存复用；
- **有效性**：响应是否非空、是否被异常截断。

> **一个被我们修正过的误区**：早期版本曾用「输出长度是否落在固定区间」做指纹，结果正品的中文四行诗（长度恒为 35 字左右）被误杀。我们改用「内容是否真正多样」作为核心信号——**长度相同不等于内容相同**，固定格式下的等长输出恰恰是真实模型的正常表现。这正是「在真品上反复校准」的价值。

### ⑦ 协议来源指纹（协议层 · 杀手锏）

这是 Tiny Owl 体系中**最难被伪造、权重最高**的维度，也是我们识破「高仿替换」的关键。

行为可以演，但响应的**协议元数据**会暴露上游真实厂商。我们采集并比对一组「厂商签名」：

| 厂商 | id 前缀 | usage 特有字段 | 结束原因 | 其它 |
| --- | --- | --- | --- | --- |
| **Anthropic** | `msg_` | `cache_creation_input_tokens`、`claude_cache_creation_*` | `end_turn`、`stop_sequence` | — |
| **OpenAI** | `chatcmpl-` | `prompt_tokens_details`、`reasoning_tokens` | `stop`、`length` | `system_fingerprint` |
| **Google** | — | `promptTokenCount`、`candidatesTokenCount` | `STOP`、`SAFETY` | — |

判定逻辑：

- 响应元数据指向**目标厂商** → 通过（强正品证据）；
- 响应元数据指向**竞品厂商**（如声称 Claude 却带 `chatcmpl-` 前缀和 OpenAI 的 `usage` 结构）→ **判定失败，认定来源冲突**；
- 元数据被完全规范化、无任何厂商特征 → 可疑（来源不可证实）。

> **为什么难伪造**：要骗过这一维度，中转站必须主动重写**每一个响应**的 id 前缀、usage 字段结构、结束原因词表、乃至流式分块格式——成本远高于「在 system prompt 里塞一句话」。绝大多数廉价中转会直接透传上游响应，从而当场暴露真实来源。这与学界对「模型替换审计」的核心判断一致：信号越接近底层、越难被提供方单方面操纵，鉴别力就越强 <sup>[1][3]</sup>。

## 评分聚合：让最强信号说话

七个维度并非平均主义。Tiny Owl 按「信号可信度」加权，把权重向最难伪造的协议层倾斜：

| 维度 | 权重 | 层级 |
| --- | --- | --- |
| 协议来源指纹 | **0.25** | 协议层 |
| 身份一致性 | 0.20 | 行为层 |
| 知识问答结果 | 0.15 | 行为层 |
| 返回协议一致性 | 0.13 | 协议层 |
| 签名指纹 | 0.12 | 统计层 |
| 响应结构 | 0.10 | 协议层 |
| 思维链痕迹 | 0.05 | 行为层 |

在加权求和之外，我们设置了一条**一票否决（硬封顶）规则**：

> 当「协议来源指纹」判定为**来源冲突**时，无论其它六个维度表现多好，综合评分被**强制封顶到 30 分以下**，并打出「疑似模型身份替换」警示。

这条规则正是为了对付「行为层伪装得天衣无缝」的高仿替换——它把行为表演的收益直接清零。无法判定（inconclusive）的维度则按比例从权重池中剔除，避免无谓拉低评分。

## 实战检验：一次真实校准

> 以下是我们用一个**正品 Claude 中转**和一个**高仿山寨中转**做的对照实测，最能说明七维体系的价值。

**测试对象 A：正品中转**（目标模型 claude-opus-4-6）

| 维度 | 结论 | 子分 |
| --- | --- | --- |
| 返回协议一致性 | 通过 | 100 |
| 响应结构 | 通过 | 100 |
| 知识问答 | 通过 | 100 |
| 身份一致性 | 通过（合规拒答，无竞品信号） | 85 |
| 思维链痕迹 | 通过 | 100 |
| 签名指纹 | 通过（内容去重率 100%） | 100 |
| **协议来源指纹** | **通过（命中 Anthropic 签名）** | **100** |
| **综合** | | **97** |

协议来源指纹捕获到了 `msg_` 前缀与 `claude_cache_creation_5_m_tokens` 字段，铁证其上游为 Anthropic。

**测试对象 B：高仿山寨**（声称 claude-opus-4-6，实际后台是 OpenAI，且刻意伪装）

这个山寨站把**行为层全部伪装到位**：自称「我是 Claude」、知识题全答对、诗作内容多样、推理痕迹完整。

| 维度 | 结论 | 子分 |
| --- | --- | --- |
| 身份一致性 | 通过（成功表演） | 100 |
| 知识问答 | 通过 | 100 |
| 思维链痕迹 | 通过 | 100 |
| 签名指纹 | 通过 | 100 |
| **协议来源指纹** | **未通过（元数据指向 OpenAI）** | **0** |
| **综合** | | **30（识破）** |

**结论一目了然**：仅看行为层，山寨站能拿到 95+ 分蒙混过关；而协议来源指纹发现它的响应带着 `chatcmpl-` 前缀和 `system_fingerprint`，铁证上游是 OpenAI，**一票否决，综合分封顶到 30，并触发身份替换警示**。

> 这就是 Tiny Owl 的底气：**行为可以演，协议不会撒谎。**

## 可选项：缓存检测模式

开启后（约增加 30 秒），Tiny Owl 会对同一与近似 prompt 发送重复请求，比较响应是否字节级一致且延迟异常偏低，判定中转站是否在用缓存「偷工减料」。关闭则完全跳过。

## 隐私优先：你的 Key，用后即焚

技术鉴别不应以牺牲安全为代价。Tiny Owl 对用户 API Key 的处理遵循严格原则：

- **加密传输**：全程 HTTPS；
- **仅存内存**：Key 只在检测任务执行期间存活于服务端内存，绑定任务 ID；
- **用后即焚**：任务结束（无论成功、失败、出错），在 `finally` 中立即从内存销毁；
- **日志掩码**：任何日志中形似密钥的字符串，一律以「首 4★★★★尾 4」掩码记录；
- **历史脱敏**：检测历史只保存脱敏结果（接口标识掩码、各维度结论），**绝不含 Key 明文**。

## 边界与诚实声明

我们相信，可信的检测从坦诚开始。

- **这不是法律审计，不保证 100% 准确。** 检测结论是低成本的技术性验证，供你参考决策，而非定性证据。
- **没有任何纯软件方法能 100% 防伪。** 学界共识是：面对一个愿意付出极高成本、逐字段重写整套响应协议的对手，纯软件检测无法提供绝对保证；真正密码学级别的保障需要可信执行环境（TEE）等厂商侧基础设施配合 <sup>[1]</sup>。Tiny Owl 能做到的，是**把作假门槛从「塞一句 system prompt」抬高到「重写整套协议元数据」**，让绝大多数掺水、降智、替换的廉价中转当场现形。
- **技术指标 ≠ 信誉保证。** 榜单排名高、检测分数高，也不等于绝对可信。我们的建议始终是：**先小额充值试用，确认稳定后再大额投入。**

本文方法学参考了以下黑盒 LLM API 审计与模型指纹研究：

1. **Are You Getting What You Pay For? Auditing Model Substitution in LLM APIs** — 系统形式化了 LLM API 中的模型替换检测问题，评估了文本分类、身份提示、基准测试、logprob 验证、TEE 等多种方法，并指出纯软件文本方法难以对抗有意作恶者。https://arxiv.org/abs/2504.04715
2. **Auditing Black-Box LLM APIs with a Rank-Based Uniformity Test (RUT)** — 提出基于秩的均匀性检验，在仅有文本输出的黑盒条件下验证 API 背后模型是否被替换。https://arxiv.org/abs/2506.06975
3. **Deceptive Model Claims in Shadow APIs** — 研究「影子 API」输出与官方 API 的一致性问题，揭示中转 / 转售场景下的模型声明欺诈风险。https://arxiv.org/html/2603.01919
4. **Log Probability Tracking of LLM APIs** — 通过追踪 API 的对数概率分布，监测所服务模型随时间发生的变更与替换。https://arxiv.org/html/2512.03816
5. **Fingerprinting LLMs via Prompt Injection** — 通过优化指纹提示，使其触发模型特有的 token 偏好，从而获得对基座模型唯一且鲁棒的指纹。https://arxiv.org/abs/2509.25448
